Sprint 1 Project Setup & Weather API FastAPI · Open-Meteo · Expo init 0/10 ▾
Goal: Dev environment configured, Open-Meteo returning real data through the Python API.
Python API (FastAPI)
Initialize FastAPI project structure with environment config
Create the FastAPI project with a clean folder structure: /app, /app/routes, /app/models, /app/services, /app/config. Set up .env file management using python-dotenv or Pydantic Settings. Include separate configs for development, staging, and production environments. Database URL and CORS origins should be environment variables from day one. Open-Meteo requires no API key, but include a WEATHER_PROVIDER env var to support future provider swaps.
APIInfra
Set up repository, branching strategy, and CI basics
Initialize Git repo. Recommended branching: main (production), develop (integration), feature/* branches. Set up a basic CI pipeline (GitHub Actions recommended) that runs linting and tests on PR. Keep it simple — just pytest + flake8/ruff. No CD yet; deployment is manual this sprint.
Infra
Implement /weather endpoint — hourly + daily from Open-Meteo by lat/lng
Create GET /weather?lat={lat}&lng={lng} endpoint. Call Open-Meteo's Forecast API to fetch hourly forecast (48 hours) and daily forecast (7 days). No API key required — Open-Meteo is free and open source. Use httpx or aiohttp for async HTTP calls. Return both hourly and daily arrays in the response. Respect Open-Meteo's fair use policy (no aggressive polling). Log request/response metadata for debugging.
API
Map Open-Meteo response to VeloVane weather model
Create Pydantic models for the VeloVane weather schema. Build a provider abstraction layer (WeatherProvider interface) with an Open-Meteo implementation that maps fields to: temperature, feelsLike, humidity, heatIndex, dewPoint, precipProbability, windSpeed, windGust, windDirection. This abstraction means if you upgrade specific regions to Tomorrow.io later, only a new provider implementation is needed — not app-wide changes. Open-Meteo returns metric by default; handle imperial conversion at the API layer.
API
Basic error handling and logging on API layer
Implement structured error responses (consistent JSON error format with error code, message, and details). Add try/except around Open-Meteo calls with specific handling for: 5xx (provider down), network timeouts, and malformed responses. Open-Meteo has no auth (no 401) but may return 429 under heavy load. Use Python logging with structured output (JSON logs recommended for production). Include request ID in all logs for tracing.
API
Deploy API to staging environment
Deploy to Railway, Render, or Fly.io — whichever you're most comfortable with. Railway is the fastest for Python; Fly.io gives more control. Confirm the /weather endpoint is accessible from the public internet. Set up environment variables in the hosting platform. Document the deploy process so it's repeatable. This is staging only — production deploy comes later.

Sprint 4 Rules Engine & Recommendation Logic Scoring · Wind direction · Unit tests 0/8 ▾
Goal: Ride window recommendation engine running server-side for any user profile + forecast combination.
Python API
Build /recommendations endpoint
Create GET /recommendations?lat={lat}&lng={lng} (authenticated). This endpoint fetches the weather forecast, loads the user's preference profile, and runs the rules engine to identify optimal ride windows. Response includes: an array of ranked windows (top 1-3), each with start_time, end_time, score, conditions summary, and plain-language reason. If no windows meet preferences, return the next-best window with the primary blocker reason.
API
Rules engine: evaluate 48hr forecast against user thresholds, score + rank windows
For each hour in the 48hr forecast, calculate a composite score based on: (1) wind speed vs user threshold (0 if exceeded, scaled 0-1 below), (2) temperature vs comfort range (1 if within, scaled penalty outside), (3) precip probability (1 if <10%, linear penalty to 0 at 80%+), (4) time of day match (bonus for preferred times, penalty for non-preferred). Group consecutive hours with scores above a threshold into "ride windows." Rank windows by average score × duration. Return the top 3.
APICritical
Wind scoring — ride-pattern-aware (out-and-back vs. fixed-route)
Two scoring modes based on ride_pattern preference. OUT-AND-BACK (default): (1) Wind speed is the primary factor with a nonlinear penalty curve — a 20mph wind is more than twice as bad as 10mph because headwind energy cost scales exponentially. (2) Crosswind exposure penalty: calculate the worst-case crosswind component for any plausible out-and-back direction. A pure crosswind scores worst (no direction choice gives a tailwind leg). (3) Wind consistency bonus: measure directional variance across hours in the window. Steady direction = bonus (easy to plan out-and-back). Shifting >90° = penalty with explicit warning. (4) Wind trend: if speed is increasing significantly during the window, flag it — rider may return into worse conditions than they left in. FIXED-ROUTE: score headwind/tailwind/crosswind per hour against stated direction (same as v2 logic). Headwind penalty scaled by speed, tailwind small bonus, crosswind neutral.
APINew v2
Dawn/dusk constraint — penalize or flag windows overlapping darkness
Windows that start or end in darkness (before civil dawn or after civil dusk) should receive a scoring penalty — not an automatic disqualification, since some cyclists ride with lights. If a top-ranked window overlaps darkness, include a safety note in the reasoning: "This window starts 20 min before sunrise — bring lights for the first portion." If the entire window is in darkness, penalize heavily but still show it if no daylight alternatives exist.
APINew v2
Plain-language reason generator (template-based, not LLM)
Build a template system that generates human-readable explanations for each recommended window. Format varies by ride pattern. OUT-AND-BACK: "7–9am: 64°F (feels like 58° on the bike), steady 12mph from the west, less than 5% rain — great conditions for an out-and-back." When wind shifts during window: "Wind shifts from W to SW during this window — plan your turnaround timing." When wind is building: "Wind picks up from 10 to 18mph over this window — start early for an easier return." FIXED-ROUTE: "7–9am: 64°F (feels like 58° on the bike), 9mph tailwind, less than 5% rain — well within your preferences." Both: When cycling feels-like delta exceeds 5°F, call it out: "It's 72° but will feel like 64° on the bike at your speed." Use plain, direct language — no marketing speak. This is NOT an LLM call — it's string templates with variable insertion.
API
No-good-window handler: next available + primary blocker reason
When no hours in the 48hr forecast meet the user's preferences, the endpoint should: (1) identify the primary blocker (e.g., "Wind exceeds your 20mph threshold all day"), (2) find the next-best window even though it doesn't fully qualify, (3) identify when conditions are expected to improve ("Tomorrow morning looks better — wind drops to 12mph"). Never return an empty recommendation — always give the cyclist something actionable, even if it's "today isn't your day."
API
Unit tests: all-day rain, high wind, wind shifts, dawn/dusk edge cases
Write pytest tests covering: (1) all-day rain at 80%+ probability → no window, blocker = "rain", (2) wind above threshold all 48 hours → no window, shows when wind drops, (3) out-and-back: steady wind scores better than shifting wind at same speed, (4) out-and-back: pure crosswind (perpendicular to all axes) scores worse than head/tail wind at same speed, (5) out-and-back: wind building from 10→22mph during window triggers trend warning, (6) out-and-back: wind shifting 90°+ during window triggers shift warning, (7) fixed-route: headwind all day but tailwind tomorrow → recommends tomorrow, (8) perfect conditions but entirely before dawn → flags darkness, (9) overnight-only favorable window → penalized but shown, (10) user with out-and-back pattern → wind scoring uses speed + consistency, (11) user with fixed-route pattern → wind scoring uses directional analysis, (12) edge case where temp range is inverted (user error). Minimum 15 test cases. These tests are your safety net before Sprint 5 builds UI on top.
APICritical
React Native / Expo
Wire up recommendation API call + temporary debug view
Add a call to /recommendations on dashboard load (after weather data loads). Display the raw JSON response in a temporary debug view — this lets you verify the engine is working correctly before investing in the polished UI next sprint. Confirm the recommendation data includes all expected fields: windows array, reasons, blocker info. This is a throwaway screen that gets replaced in Sprint 5.
Mobile
Done when: Rules engine returns accurate, preference-aware ride windows with ride-pattern-specific wind scoring (out-and-back: speed + consistency + crosswind; fixed-route: directional). Dawn/dusk constraints applied. Covered by 15+ unit tests.
Senior Dev Flag: Recommendation scoring logic can get complex fast. Sanity-check algorithm design before building UI on top.

Sprint 7 GPS Inference Engine Background location · Ride detection 0/8 ▾
Goal: Passive ride detection running on-device. Inferred rides updating preference profile.
React Native / Expo
Implement background location tracking (Expo Location)
Set up Expo's background location task using TaskManager. This runs even when the app is in the background. MobileCritical
On-device inference: detect 10–25mph sustained for 30+ min → cycling
Build the ride detection algorithm entirely on-device (no API calls during detection). Logic: if location updates show sustained movement at 10–25 mph for 30+ consecutive minutes, classify as a likely cycling session. Filter out: driving (>30mph sustained), walking (<8mph), and stationary periods with GPS drift. Use conservative thresholds — false negatives (missing a ride) are better than false positives (detecting a car trip as cycling). The algorithm should run in the background task with minimal battery impact.
MobileCritical
Record inferred ride: datetime, duration, weather conditions at time
When a likely ride is detected, record: start_time, end_time, duration_minutes, and the weather conditions at the ride's location during the ride (fetch from cached weather data or make a lightweight API call). Store as a local event object. This gives the AI engine a record of what conditions the user actually rode in — which is the signal for learning their true preferences (vs. what they said in onboarding).

Location permission request with cyclist-specific explanation

Include a "Learn more" link to your privacy policy. Offer an explicit "Skip for now" option — never pressure the user. If they skip, the app works fine without inference — onboarding preferences are the baseline forever.

Store inferred ride events in a local queue (AsyncStorage is fine for this). When the device has connectivity, batch-sync the queue to the /inference API endpoint. Handle the case where sync fails (network issue) — keep the events in the queue and retry on next connectivity event. Clear synced events from the local queue after successful API confirmation. This offline-first approach means ride detection works even without internet.
Mobile
Python API
/inference endpoint: receives inferred ride events
Create POST /inference that accepts a batch of inferred ride events from the device. Validate each event: reasonable timestamps, plausible duration (30 min - 8 hours), and non-duplicate (check against existing events). Store in an inferred_rides table: user_id, start_time, end_time, duration, weather_conditions (JSON), confidence (initially "inferred" — upgraded to "confirmed" if user taps Yes on the nudge). Index by user_id and start_time for efficient querying.
API
Preference update logic (additive model, min 10 rides before adjustment)
Build the behavioral learning algorithm. Approach: after a user has 10+ inferred/confirmed rides, calculate the actual range of conditions they've ridden in. If they consistently ride in 18-20mph wind despite setting a 15mph threshold, gradually adjust the inferred threshold upward. Key rules: (1) never lower a threshold below what the user set in onboarding (that's their stated floor), (2) only raise thresholds based on actual behavior, (3) require 10 rides minimum for any adjustment, (4) weight confirmed rides (nudge = Yes) 3x higher than inferred-only rides.
API
Onboarding preferences remain baseline until sufficient data
Critical safeguard: onboarding preferences are the baseline and should never be overwritten by behavioral inference. Instead, maintain two layers: "stated" (from onboarding) and "learned" (from behavior). The rules engine uses the learned values when available, but the user can always revert to stated values. The My Profile screen (Sprint 8) will show both layers clearly. This protects against inference errors degrading the experience.
